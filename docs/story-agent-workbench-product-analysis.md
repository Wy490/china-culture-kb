# Story Agent 工作台化升级产品分析

> 版本：v0.1  
> 日期：2026-06-09  
> 目标项目：`china-culture-kb`  
> 关联文档：`docs/product-document.md`、`开发文档/story_agent_platform_full_development_plan.md`、`开发文档/story-agent-creative-reference-memory-mosaic-dev-doc.md`  
> 分析目标：评估“知识库折叠、多知识库并存、故事列表收敛、单段修改、导出保存”等想法是否符合用户习惯，识别风险，并补足后续设计要点。

---

## 1. 结论摘要

你的想法整体符合用户习惯，而且方向是对的。

它代表产品正在从：

```text
单次生成工具
```

升级为：

```text
故事创作工作台
```

这个升级对真实用户更友好，因为真实用户不会只满足于：

- 只浏览一次知识库；
- 只生成一次故事；
- 生成后不能改；
- 只能复制结果走人。

他们更常见的需求是：

- 先找到知识来源；
- 生成一个初稿；
- 挑出不满意的局部继续修改；
- 反复比较多个版本；
- 导出本地沉淀；
- 后续再回来继续改。

所以这次你提出的方向，本质上是在补齐“创作闭环”。

---

## 2. 你的想法是否符合用户习惯

### 2.1 知识库折叠，用户自行展开

符合习惯，尤其是你明确说后面会有很多不同类型知识库。

原因：

- 首页信息密度会越来越高，不能长期把一个知识库的全部入口平铺在首页；
- 用户更容易理解“先选知识库，再选库内维度”；
- 未来如果有中国文化、警察故事、小说改编、创作参考等多个知识库，折叠或卡片入口几乎是必需的；
- 这更接近工作台型产品，而不是单一主题网站。

结论：

```text
符合习惯，建议保留。
```

### 2.2 最近生成故事不全量展示

非常符合习惯。

原因：

- 首页上的“最近生成故事”本质是预览，不应承担完整项目管理职责；
- 故事一多之后，全量平铺会挤压首页主任务；
- 用户在首页只需要“看最近发生了什么”，不需要立刻管理全部资产。

结论：

```text
符合习惯，应改成少量预览 + 查看全部。
```

### 2.3 支持单独修改某一段

这是最符合真实创作习惯的一项，也是产品价值最大的方向之一。

真实用户很少会说：

```text
整篇都不行，全部重来。
```

更常见的是：

- 第 3 段冲突不够强；
- 某场对白太弱；
- 某个 `gears_segment` 太长；
- 某场视觉描述不对；
- 某段风格不符合要的成片类型。

结论：

```text
非常符合习惯，但它不是小功能，而是产品能力升级。
```

### 2.4 支持导入本地保存

符合习惯，而且是一个很自然的预期。

因为一旦用户把故事当成生产资料，他们就会默认：

- 能下载；
- 能存档；
- 能发给别人；
- 能进入别的流程；
- 能以后再打开。

结论：

```text
符合习惯，应当尽快具备。
```

## 3. 主要风险

### 3.1 信息架构风险

如果你只是把“中国文化知识库”折叠起来，但没有重新定义首页结构，后面会出现：

- 用户看不懂首页到底是“知识库首页”还是“故事工作台”；
- 多个知识库加入后，首页会继续堆叠；
- “知识库”“故事项目”“生成入口”三者关系不清。

风险本质：

```text
不是折叠本身有问题，而是首页角色需要重定义。
```

### 3.2 故事列表失控风险

“最近生成故事”如果只是少展示几个，但没有独立项目页，问题只是被推迟，不是解决。

后续风险：

- 用户找不到旧故事；
- 没有搜索和筛选；
- 无法区分草稿、已编辑、已导出、已定稿；
- 相似故事过多时很难管理。

### 3.3 单段修改带来的一致性风险

这是最关键的风险。

如果允许改单段，就会出现：

- `full_text` 和 `scene_breakdown` 不一致；
- `scene_breakdown` 和 `gears_segments` 不一致；
- 单段改好了，但整篇叙事节奏被破坏；
- 后续整篇重生成覆盖用户修改；
- 不同版本之间无法追溯。

本质问题：

```text
单段修改不是纯前端编辑，而是“局部重生成 + 版本管理 + 联动更新”。
```

### 3.4 导出边界不清风险

如果只说“导出”，但不定义导出对象，用户体验会混乱。

需要提前明确：

- 导出正文还是完整项目；
- 导出当前版本还是原始版本；
- 导出 Markdown / JSON / GEARS segments 中哪些；
- 是否包含用户局部修改后的内容；
- 是否保留结构元数据。

### 3.5 多知识库扩展风险

现在中国文化知识库用“省份”作为二级维度非常合理，但未来其他知识库未必适合省份结构。

例如：

- 警察故事更可能按案件类型、警种、主题；
- 小说改编更可能按原作、角色、章节；
- 创作参考库更可能按风格包、成片类型、叙事结构。

风险：

```text
如果把“省份二级菜单”写死成知识库通用模式，未来会很难扩展。
```

### 3.6 成本和性能风险

单段修改意味着：

- 更频繁的生成请求；
- 更细粒度的写盘；
- 更复杂的故事数据结构；
- 更多状态同步；
- 更多历史版本文件。

如果后续接入真实模型 API，单段重生成虽然比整篇便宜，但调用次数会显著上升。

### 3.7 用户预期管理风险

如果你让用户“直接自由编辑全文”，他们会自然期待：

- 编辑完以后所有结构字段自动同步；
- 再次生成不会覆盖改动；
- AI 能理解他们的手工改法；
- 导出的内容一定和界面所见完全一致。

这会让系统复杂度瞬间上升。

所以第一阶段不能把“自由编辑”讲得太满。

## 4. 最值得补充的产品能力

### 4.1 故事项目概念

你现在系统里更多是“生成结果”。

但支持局部修改后，必须升级为：

```text
故事项目
```

一个项目至少包含：

- 项目 ID；
- 来源知识库 / 条目；
- 当前故事版本；
- 原始生成版本；
- 最近修改时间；
- 修改状态；
- 导出状态。

没有“故事项目”这个概念，后续管理会很快失控。

### 4.2 版本管理

只要有局部修改，就必须有版本。

至少要分：

- 原始生成版；
- 当前工作版；
- 历史版本；
- 是否已导出；
- 是否已定稿。

### 4.3 修改粒度

这是最需要提前定清楚的地方。

建议顺序：

1. 先支持 `scene` 级重生成；
2. 再支持 `gears_segment` 级重生成；
3. 最后才考虑全文自由编辑。

原因：

- `scene` 和 `segment` 是当前数据结构中最稳定的编辑单位；
- 可以保持结构化；
- 更容易联动更新；
- 更容易做质量校验。

### 4.4 修改方式模板化

不要一开始只给用户一个空输入框说“写提示词改这段”。

更符合习惯的方式是提供常用操作：

- 重写这一场；
- 缩短这一场；
- 延长这一场；
- 增强冲突；
- 增加对白；
- 调整成更纪实；
- 调整成更影视化；
- 只重写这一段 `gears_segment`。

这样能降低门槛，也更稳定。

### 4.5 故事项目库页面

首页不能承载全部故事管理。

你需要单独页面承接：

- 我的故事；
- 故事项目库；
- 故事历史。

这个页面至少要有：

- 搜索；
- 筛选；
- 排序；
- 状态标记；
- 进入编辑；
- 导出。

### 4.6 导出格式分层

建议不要只做一个“导出”按钮。

应拆成：

- 导出故事正文 `Markdown`
- 导出完整项目 `JSON`
- 导出 `GEARS segments JSON`

这样更符合不同用户预期。

### 4.7 首页角色重定义

首页后续不应该再是：

```text
知识库网格 + 最近故事全铺开
```

而应该是：

```text
知识库入口
+ 故事项目入口
+ 故事生成入口
```

也就是从“内容首页”升级为“工作台首页”。

## 5. 哪些点最符合真实用户习惯

从优先级看，最贴近真实创作习惯的功能顺序是：

1. 首页只展示少量最近故事，另设“查看全部故事”
2. 中国文化知识库折叠为统一入口
3. 增加故事项目库页面
4. 支持单场景 / 单段重生成
5. 支持导出 Markdown / JSON / GEARS
6. 再考虑自由文本编辑

原因很简单：

- 1 到 3 解决的是“找得到、管得住”；
- 4 解决的是“改得动”；
- 5 解决的是“带得走”；
- 6 才是“完全编辑器化”。

---

## 6. 不建议现在就做的事

### 6.1 不建议先做全文任意编辑

原因：

- 会破坏结构一致性；
- 难以与生成链路联动；
- 很快会需要复杂编辑器和 diff 机制；
- 会让版本管理和导出变复杂。

### 6.2 不建议首页继续承载所有功能

如果首页继续放：

- 省份知识库；
- 最近故事大列表；
- 未来多个知识库；
- 未来多个项目入口；

那首页会越来越重。

### 6.3 不建议把“省份二级菜单”做成通用知识库模型

它适用于中国文化知识库，但不应成为所有知识库的通用结构。

---

## 7. 推荐的产品结构升级方向

建议把产品结构从现在的：

```text
首页
  ├─ 中国文化知识库
  ├─ 搜索
  └─ 故事生成
```

升级为：

```text
故事 Agent 工作台
  ├─ 知识库
  │   ├─ 中国文化知识库
  │   │   └─ 省份 / 类型 / 主题
  │   ├─ 警察故事知识库（后续）
  │   ├─ 小说改编知识库（后续）
  │   └─ 创作参考库（后续）
  ├─ 故事项目库
  ├─ 故事生成工坊
  └─ 导出 / GEARS 对接
```

这个结构更稳，也更符合你后面会扩张多知识库、多领域、多项目的现实。

## 8. 推荐的 MVP 切分

### MVP-1：信息架构重构

- 首页改为工作台入口页；
- 中国文化知识库默认折叠；
- 最近故事只展示 3 到 6 条；
- 新增“查看全部故事”入口。

### MVP-2：故事项目库

- 新增故事列表页；
- 搜索、筛选、排序；
- 状态标记：草稿 / 已编辑 / 已导出。

### MVP-3：局部重生成

- 只支持 `scene` 级重生成；
- 支持几种结构化修改意图；
- 修改后同步更新相关 `gears_segments`；
- 生成历史记录。

### MVP-4：导出

- 导出 Markdown；
- 导出完整 JSON；
- 导出 GEARS segments。

### MVP-5：高级编辑

- `gears_segment` 级重生成；
- 更细粒度版本对比；
- 再讨论是否开放全文自由编辑。

---

## 9. 对你这次想法的最终判断

你的想法是成立的，而且是产品从“能生成”走向“能创作、能管理、能沉淀”的关键一步。

最重要的不是“要不要做”，而是“按什么顺序做”。

最稳妥的方向是：

```text
先工作台化
  → 再项目化
  → 再局部重生成
  → 再导出
  → 最后再做更自由的全文编辑
```

如果顺序反过来，比如先上全文编辑或先上复杂导出，后面会反复返工。

---

## 10. 给后续执行文档的约束

执行层面建议遵守以下约束：

1. 不改写知识库事实条目 `data/provinces`
2. 故事修改只作用于生成内容和项目版本
3. 先做局部重生成，不做全文任意编辑
4. 所有修改都进入版本记录
5. 导出以当前工作版本为准
6. 首页只做入口和概览，不承担完整项目管理
7. “省份”只作为中国文化知识库的二级维度，不抽象为所有知识库的统一模式
